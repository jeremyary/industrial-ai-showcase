# HIL Approval Drawer — Design Specification

> [!NOTE]
> This project was developed with assistance from AI tools.

This document specifies the Human-in-the-Loop (HIL) approval drawer for the
Showcase Console. It defines what the operator sees when an AI agent proposes
a state-modifying action, what information they need to make a decision, and
how that decision is recorded. Implementation lands in Phase 3; this spec
ships at the end of Phase 2 to lock the contract.

## Design Invariant

**HIL gating operates on the GitOps / PR-open path only.** It is never inline
in the serving-time robot command flow. The governance layer cannot add latency
to 10 Hz+ VLA inference. Violating this invariant breaks the architecture's
credibility with technical audiences.

In practice: the agent proposes a change → the change becomes a pull request →
the operator reviews and approves in the drawer → the PR merges → Argo CD
reconciles. The robot's real-time control loop is untouched.

## When the Drawer Opens

The drawer opens when the LangGraph orchestrator emits a tool call classified
as **state-modifying**. The classification is per-MCP-tool, declared in the
tool's registration metadata:

| MCP Server | Tool | Classification | Example |
|------------|------|----------------|---------|
| `mcp-fleet` | `get_fleet_status` | Read-only | Query robot positions and mission state |
| `mcp-fleet` | `override_mission_params` | State-modifying | Change speed limit for a factory |
| `mcp-fleet` | `propose_fleet_intervention` | State-modifying | Reassign robots between zones |
| `mcp-isaac-sim` | `list_scenes` | Read-only | Enumerate available simulation scenes |
| `mcp-isaac-sim` | `launch_sim_run` | State-modifying | Start a simulation experiment |
| `mcp-mlflow` | `query_experiments` | Read-only | Search experiment results |
| `mcp-mlflow` | `promote_model_version` | State-modifying | Promote a model to production stage |
| `mcp-mlflow` | `register_model` | State-modifying | Register a new model version |

Read-only tool calls pass through without operator intervention. The agent
can freely read fleet state, query MLflow experiments, and browse Nucleus
assets. Only actions that would change cluster state, launch workloads, or
modify model registrations require approval.

## What the Drawer Shows

The drawer is a slide-out panel on the right side of the Console. It contains
five sections, top to bottom:

### 1. Proposed Action Summary

A human-readable description of what the agent wants to do, generated by
the LangGraph orchestrator alongside the tool call.

```
Agent proposes: Promote vla-warehouse-v1.4 to production for Factory A

Reason: Training run mlflow://experiments/12/runs/abc123 shows 14% improvement
in pick-success rate over v1.3 baseline across 200 evaluation episodes.
```

This is the agent's own explanation. It is not editable by the operator —
it's evidence of the agent's reasoning chain.

### 2. Proposed Diff

The concrete change that would be applied if approved. For the agent-opens-a-PR
pattern, this is the actual git diff of the PR the agent would create:

```diff
--- a/infrastructure/gitops/apps/workloads/factory-a/policy-version.yaml
+++ b/infrastructure/gitops/apps/workloads/factory-a/policy-version.yaml
@@ -4,5 +4,5 @@ metadata:
   namespace: warehouse-edge
 data:
-  POLICY_VERSION: "vla-warehouse-v1.3"
+  POLICY_VERSION: "vla-warehouse-v1.4"
```

For actions that don't map cleanly to a git diff (e.g., launching a simulation),
the diff section shows a structured representation of the API call parameters.

### 3. Blast-Radius Analysis

A machine-generated summary of what the proposed change affects:

| Dimension | Value |
|-----------|-------|
| Affected factories | Factory A (companion cluster) |
| Affected robots | 1 (G1 unit currently running v1.3) |
| Rollback path | `git revert` + Argo sync (measured: <20s) |
| Service disruption | None — hot-swap, no pod restart required |

The blast radius is computed by the governance layer based on the tool call
target. For fleet interventions, it queries `mcp-fleet` (read-only) to
determine which factories and robots are in scope. For model promotions,
it queries `mcp-mlflow` for downstream consumers.

### 4. Context Trail

The read-only MCP tool calls the agent made to arrive at this proposal. This
answers "how did the agent decide this?" without requiring the operator to
read the full agent trace.

```
1. mcp-mlflow: query_experiments("vla-training") → 3 runs found
2. mcp-mlflow: get_run("abc123") → pick_success=0.87, baseline=0.76
3. mcp-fleet:  get_fleet_status("factory-a") → 1 robot, policy v1.3, no active anomalies
4. mcp-fleet:  get_anomaly_history("factory-a", hours=24) → 0 anomalies in last 24h
```

Each entry links to the source system (MLflow run URL, fleet status snapshot)
so the operator can verify the agent's claims independently.

### 5. Guardrail Outcomes

Results from the Llama Stack safety layer's checks on this action:

| Check | Result | Detail |
|-------|--------|--------|
| PII scan | Pass | No PII detected in agent reasoning or tool call parameters |
| Safety guardrails | Pass | Action does not violate fleet safety constraints |
| Policy evaluation | Score: 0.87 vs. incumbent 0.76 | TrustyAI evaluation of proposed policy against current baseline |

A guardrail failure blocks the action from being approvable. The operator sees
the failure reason and can instruct the agent to revise its approach.

## Operator Actions

The drawer footer presents two actions:

- **Approve** — the PR is merged (or the API call is executed), and the
  approval event is written to the audit store. The drawer closes and a
  toast notification confirms the action.
- **Reject** — the PR is closed (or the tool call is discarded), with an
  optional operator-provided reason. The rejection and reason are written
  to the audit store. The agent receives the rejection and can propose an
  alternative.

There is no "Approve with modifications" — the operator either approves the
agent's exact proposal or rejects it. If the proposal needs changes, the
operator rejects with a reason, and the agent incorporates the feedback
into a new proposal. This keeps the audit trail clean: every approved action
is exactly what the agent proposed.

## Audit Record

Every HIL decision (approve or reject) produces an immutable audit record:

```json
{
  "timestamp": "2026-05-01T14:32:07Z",
  "action_id": "hil-2026-05-01-143207-abc",
  "agent_session": "langgraph-session-xyz",
  "tool_call": {
    "mcp_server": "mcp-mlflow",
    "tool": "promote_model_version",
    "parameters": {
      "model": "vla-warehouse",
      "version": "v1.4",
      "stage": "production"
    }
  },
  "classification": "state-modifying",
  "guardrail_results": {
    "pii_scan": "pass",
    "safety": "pass",
    "policy_eval_score": 0.87
  },
  "decision": "approved",
  "operator_identity": "jary@redhat.com",
  "context_trail_hash": "sha256:abcdef...",
  "pr_url": "https://github.com/org/repo/pull/42"
}
```

**Identity binding:** The operator's identity comes from OpenShift's OAuth
token. For regulated environments, this extends to CAC/PIV certificate-bound
sessions — the approval is tied to a cryptographic identity, not just a
username. The specific identity provider configuration is deployment-specific.

**Storage:** Audit records are written to an append-only store. The
implementation choice (Kafka topic with compaction disabled, S3 bucket with
object lock, or a dedicated audit database) is deferred to Phase 3
implementation. The record schema above is the contract.

## Agent-Opens-a-PR Pattern

The most common HIL flow for this architecture is the agent-opens-a-PR pattern:

```
Operator asks: "Factory A's pick rate dropped — investigate and fix"
    ↓
LangGraph agent investigates (read-only MCP calls, no HIL needed):
  - Queries fleet telemetry for anomaly patterns
  - Queries MLflow for recent training runs
  - Queries Isaac Sim for recent scenario results
    ↓
Agent proposes: "Promote v1.4 — training run shows improvement"
    ↓
Agent opens a PR to infrastructure/gitops/ with the policy version change
    ↓
HIL drawer opens in the Console showing the PR diff
    ↓
Operator reviews → Approves
    ↓
PR merges → Argo CD syncs → Factory A receives v1.4
    ↓
Audit record written with operator identity, PR URL, guardrail results
```

This pattern reframes "LLM touching production" as "LLM participating in the
code review process you already trust." The agent never calls the Kubernetes
API directly. Every state change flows through Git, through review, through
Argo CD — the same path a human engineer would take.

## What the Drawer Does NOT Do

- **It does not gate real-time robot control.** VLA inference runs at the spoke
  on a 10 Hz loop. The HIL layer has no role in that path.
- **It does not replace GitOps.** Approval in the drawer merges a PR.
  Argo CD reconciles. There is no kubectl/oc apply behind the scenes.
- **It does not make autonomous decisions.** Even if every guardrail passes
  and the eval score is high, the operator must explicitly approve. There is
  no auto-approve threshold. (Auto-approve for low-risk read-only actions is
  a Phase 4+ consideration, gated on customer feedback.)
- **It does not display raw LLM tokens.** The agent's reasoning is summarized,
  not streamed token-by-token. The full trace is available in MLflow/observability
  for post-hoc analysis.

## Console Integration

The HIL drawer is a feature of the Console's expert and evaluator audience
modes. It appears in two contexts:

1. **Agentic panel** — when the operator issues a natural-language command
   and the agent's plan includes a state-modifying tool call, the drawer
   opens inline next to the agent's plan visualization.
2. **Fleet view** — when the Fleet Manager's agentic path (anomaly
   investigation → proposed intervention) produces a HIL-gated action,
   a notification badge appears on the Fleet view and clicking it opens
   the drawer.

Pending approvals persist across page reloads (backed by the backend's
agent session state). If the operator navigates away and returns, pending
HIL requests are still visible.

## Behavioral Decisions

**Concurrent approvals:** The drawer presents proposals one at a time. If the
agent proposes multiple state-modifying actions in sequence (e.g., promote
policy for Factory A, then Factory B), each gets its own drawer instance with
its own audit record. The agent blocks on the current proposal before
advancing to the next. This is simpler, produces cleaner audit trails, and
avoids the cognitive load of batch-approving actions with different blast
radii.

**No timeout.** Pending HIL requests persist indefinitely. The operator can
navigate away, close the browser, and return hours later — the request is
still there. There is no automatic abandonment or escalation. The agent
session is paused at the HIL gate until the operator acts. This is the
correct posture for production safety-critical operations: if an operator
walks away from a proposed fleet intervention, the safe behavior is to
wait, not to assume intent.

**Multi-operator visibility:** In production, pending HIL requests should
be visible to any operator with the appropriate RBAC role, not just the
session initiator. A SOC shift change should not orphan pending approvals.
This is documented as the architectural expectation for production
deployments. The Showcase Console implementation is single-operator for
simplicity — RBAC-gated multi-operator notification is out of scope for
the demo but should be called out during deep-dive conversations.

## Drawer Layout

The drawer defaults to a right-side slide-out panel (~480px). For content
that requires more space — large diffs, complex blast-radius tables, long
context trails — the drawer supports an **expand-to-full-view** mode:

- A toggle in the drawer header switches between panel and full-page view
- Full-page view uses the entire content area, replacing the current view
  behind it (not a modal overlay — the operator can still use the sidebar
  nav to exit)
- The diff section in particular benefits from full-width rendering to
  avoid line wrapping on long YAML paths
- The operator's panel/full-page preference persists across sessions
  (localStorage)
- Keyboard shortcut: `Escape` returns to panel view from full-page
